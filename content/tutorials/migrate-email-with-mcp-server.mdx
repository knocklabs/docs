Storing email templates in Knock comes with [many advantages](/template-editor/overview#frequently-asked-questions). This tutorial shows how to use an AI agent connected to Knock's [MCP server](/ai/mcp-server) to migrate existing email templates into Knock, whether they come from an email service provider or your own system.

The migration runs as two prompts. In the first, your agent analyzes your templates and produces a migration plan for you to review. In the second, it creates the resources in Knock, commits them, runs test sends, and verifies the rendered output against your originals.

## How it works

Two roles are involved in the migration, and understanding the division of labor helps you follow along as your agent works:

- **Your agent.** The agentic client you run (for example, Claude Code or Cursor). It reads your template files, decomposes them into [layouts](/integrations/email/layouts), [partials](/template-editor/partials), and per-template content, and orchestrates the migration. Agents that support subagents or parallel tasks can fan out the per-template analysis, so large template sets are not a problem.
- **The Knock MCP server.** Your agent uses it to create, verify, and commit resources directly in your Knock account.

## Prerequisites

- A <a href="https://dashboard.knock.app/" target="_blank" rel="noopener">Knock account</a>.
- An agentic, MCP-compatible client such as Claude Code or Cursor, connected to the Knock MCP server. See the [get started](/ai/mcp-server#get-started) section of the MCP server docs for setup instructions.
- Your email template files, available on the machine where your agent runs. Knock supports HTML and [MJML](/integrations/email/mjml) templates natively. If your templates are in another format (such as React Email), export or convert them to HTML first.

### MCP capabilities

When you authenticate with the Knock MCP server, a consent screen prompts you to select which capabilities to grant. Enable the following for this tutorial.

| Capability       | Requirement | Purpose                                                                                |
| ---------------- | ----------- | -------------------------------------------------------------------------------------- |
| Read resources   | Required    | Verify created resources against the migration plan                                    |
| Manage resources | Required    | Commit changes and run test sends via the Management API                               |
| Knock agent      | Required    | Create email layouts, partials, and workflows with email templates                     |
| Debug            | Recommended | Fetch rendered message output so your agent can verify fidelity against your originals |
| Manage data      | Recommended | Create a test user to receive test emails                                              |
| Documentation    | Optional    | Look up Knock concepts during the migration                                            |

## Migration steps

The migration runs as two prompts, provided to your agent in order. We recommend leveraging a thinking model with a high effort level in order to produce a high-quality plan and execute it effectively.

<Steps titleSize="h3">
  <Step title="Analyze your templates and build a migration plan">
    This prompt has your agent read every template and produce a single `migration-plan.md` file containing the full decomposition: layouts, partials, per-template content, and variable mappings. The plan is the durable artifact of the migration. It is reviewable, additive across runs, and lets your agent resume an interrupted migration without re-reading the originals. **Replace the placeholder path before running the prompt.**

    **Prompt:**

    ```markdown title="Prompt 1: analyze your templates and build a migration plan"
    I need to migrate my existing email templates to Knock. In this first phase, analyze my legacy templates and produce a migration plan. Do not create any resources in Knock yet.

    My template files are located at: [FILL IN PATH TO YOUR TEMPLATE FILES]

    If you support subagents or parallel tasks, delegate the per-template analysis across subagents and consolidate their results. Otherwise, analyze the templates in batches small enough to preserve full fidelity. Never summarize or truncate template HTML.

    Complete the following analysis:

    1. **Inventory.** List every template file found, its filename, and its purpose based on its content.

    2. **Choose a format.** Knock supports both HTML and MJML templates.
       - If my templates are HTML, keep them as HTML. The migrated templates must be pixel-perfect replicas of the originals.
       - If my templates are MJML, keep them as MJML. Note that MJML templates require an MJML email layout.
       - Record the chosen format in the plan. Ask me before converting between formats.

    3. **Extract email layouts.** Identify the shared structural wrapper in each template: the <html>, <head>, and <body> scaffolding, CSS from the <head>, and any header and footer that repeat across templates. Consolidate identical wrappers into a single layout. Mark where template-specific content belongs with {{ content }}. Assign each layout a kebab-case key, and use the key "default" for the layout shared by the most templates.

    4. **Extract partials.** Identify reusable blocks that appear across multiple templates, such as buttons, signature blocks, and social links. Partials must be plain HTML, even when templates are MJML. Assign each partial a kebab-case key and convert dynamic values to Liquid variables.

    5. **Extract template content.** For each template, capture the remaining template-specific content exactly, preserving inline styles. Replace extracted partials with {% render 'partial-key' %}. Convert dynamic variables to Liquid syntax, using Knock's Liquid helpers where appropriate. Record the subject line, the layout key, the partials referenced, and every variable used.

    Write the complete analysis to a file named "migration-plan.md" with these sections: Overview (a table of every template with its proposed workflow key, layout key, and partials), Layouts, Partials, and Templates. Include the full extracted HTML or MJML for every item. The plan must contain everything needed to create the resources without re-reading the original files.

    If "migration-plan.md" already exists from a previous run, add to it rather than starting over.

    When the plan is complete, stop and ask me to review it before creating anything in Knock.
    ```

  </Step>
  <Step title="Review the plan">
    Before moving on, check that the layout consolidation looks right (templates grouped under the correct wrapper), the partial boundaries make sense, and the variable mappings match the data you will send when triggering workflows.
  </Step>
  <Step title="Create, commit, and verify your resources in Knock">
    This prompt has your agent create resources, verify them against the plan, commit them to the development environment, then run each workflow with test data and compare the rendered output to the original template, fixing and re-running until they match. **Replace the test email placeholder before running the prompt.** Test sends deliver real emails through your configured email provider to that address.

    **Prompt:**

    ```markdown title="Prompt 2: create, commit, and verify your resources in Knock"
    I have reviewed and approved the migration plan in "migration-plan.md". Execute the migration using the Knock MCP server. Reference the Knock documentation as needed.

    Send test emails to: [FILL IN YOUR TEST EMAIL ADDRESS]

    Work through the following phases in order:

    1. **Create resources with the Knock agent.** Use the Knock agent tool to create the resources from the migration plan, in this order: partials first, then email layouts, then one workflow per template, each with a single email step. Pass the exact HTML or MJML content from the plan; do not restyle or restructure it. Give each workflow the kebab-case key, name, and subject line from the plan. Reuse the same agent session for related follow-up fixes.

    2. **Verify against the plan.** Read back every created resource using the Management API and compare it to the migration plan. Confirm that every layout, partial, and workflow exists, that each workflow uses the correct layout, and that each template references the correct partials. Fix any discrepancies before continuing.

    3. **Commit.** Commit all changes to the development environment with a descriptive commit message.

    4. **Test each workflow.** Create a test user with my email address if one does not exist. Then, for each workflow:
       - Generate valid test data based on the variables in the plan.
       - Run the workflow in the development environment via the Management API, with the test user as the recipient.
       - Fetch the rendered message content and compare it to the original template content in the plan. Check structure, copy, links, and styling.
       - If the rendered output diverges, fix the resource with a follow-up Knock agent session, commit, and run the test again. Repeat until the output matches.
       - If you cannot fetch rendered message content, ask me to confirm each email in my inbox instead.

    5. **Report.** Produce a final summary: every workflow created and tested, the test data used, any fixes applied during verification, and anything that needs my review. Remind me to check the test emails in my inbox before promoting anything to production.
    ```

  </Step>
</Steps>

## Troubleshooting

- **A rendered email does not match the original.** Point your agent at the specific divergence and have it fix the resource, then re-commit and re-run the test.
- **The migration was interrupted.** `migration-plan.md` is the source of truth. Re-run prompt 2 and instruct your agent to first check which resources already exist in Knock and continue from where it stopped.
- **Your templates use provider-specific syntax.** Templates exported from providers often embed their own variable syntax (for example, SendGrid substitution tags, Braze Liquid, or Mailchimp merge tags). The analysis prompt converts dynamic values to [Knock's Liquid syntax](/template-editor/reference-liquid-helpers); if your provider uses unusual syntax, name it in the prompt so your agent knows what to look for.
- **Tool calls fail with permission errors.** Reconnect to the MCP server and confirm the capabilities from the table above are enabled on the consent screen.

## Frequently asked questions

<AccordionGroup>
  <Accordion title="Can I migrate React Email or other component-based templates?">
    Yes. Render or export them to HTML first, then run the migration on the HTML
    output. Knock stores templates as HTML or MJML.
  </Accordion>
  <Accordion title="How many templates can I migrate at once?">
    There is no hard limit. Agents that support subagents can analyze templates
    in parallel, and the plan file is additive, so you can run the analysis
    prompt across multiple sessions if needed.
  </Accordion>
</AccordionGroup>
