## Knock agent

The [Knock agent](/ai/agent) is a conversational assistant built into the Knock dashboard. It can do anything you'd normally do in the dashboard, with full context for the resource you're viewing and your active environment.

<Image
  src="/images/ai/knock-agent.png"
  alt="The Knock agent in the dashboard"
  width={1366}
  height={888}
  className="mx-auto"
/>

Use the Knock agent to:

- Build workflows, broadcasts, partials, email layouts, audiences, message types, and guides through conversation.
- Inspect existing resources and ask questions about how your messaging is configured.
- Pull in account-level company context and custom instructions so responses stay on-brand.

The Knock agent does not consume AI credits and is configured under **Settings** > **AI settings**.

## Agent function

The [agent function](/ai/agent-function) is a workflow step that runs a prompt on an AI model of your choice and makes the response available in the rest of your workflow run. Use it to bring AI-powered context into the notifications you send.

<Image
  src="/images/designing-workflows/agent-function.png"
  alt="A workflow with an agent step"
  width={1498}
  height={812}
  className="mx-auto"
/>

Common use cases include:

- **Enrich recipient data.** Use user and tenant properties to infer market, persona, or use case.
- **Summarize batched activity.** Distill a batch of heterogeneous events into a concise digest summary.
- **Classify or route triggers.** Tag a sign-up, support ticket, or comment so downstream steps can branch on it.

The agent function consumes AI credits and is configured per workflow step. See the [full reference](/designing-workflows/ai-agent-function) for prompt and response format details.

## MCP server

**Recommended for external clients and your own apps.** The [MCP server](/ai/mcp-server) at `mcp.knock.app/mcp` exposes Knock primitives to LLMs and AI agents via the Model Context Protocol. Reach for the MCP server when you need Knock as tools inside Claude Desktop, ChatGPT, or any MCP-compatible client, or when you're building a product that needs Knock primitives behind an LLM.

## CLI and skills

**Recommended when you're working in an IDE on a Knock-aware codebase.** The [Knock CLI](/developer-tools/knock-cli) gives a coding agent the local tools it needs to read and write Knock resources from your codebase, and [skills](/ai/cli-and-skills) give the agent the instructions it needs to use those tools correctly. Reach for this combination when you're editing code in Cursor, Claude Code, or Copilot and want the agent to manage workflows, templates, and other resources alongside your application code.

Knock publishes an open-source skills package at <a href="https://github.com/knocklabs/skills" target="_blank" rel="noopener noreferrer">github.com/knocklabs/skills</a>.

## CLI and skills vs. MCP server

The two primitives are complementary. Most teams use both: the CLI and skills inside the IDE for codebase-aware tasks, and the MCP server for ad-hoc operations and product integrations. Here's how they split:

| Question                                             | Reach for                            |
| ---------------------------------------------------- | ------------------------------------ |
| Are you working in an IDE on a Knock-aware codebase? | [CLI and skills](/ai/cli-and-skills) |
| Are you using an external GUI like Claude Desktop?   | [MCP server](/ai/mcp-server)         |
| Are you embedding Knock tools in your own product?   | [MCP server](/ai/mcp-server)         |
| Do you need repeatable, codified CLI workflows?      | [CLI and skills](/ai/cli-and-skills) |
| Do you need ad-hoc natural-language operations?      | [MCP server](/ai/mcp-server)         |
